## 第一部分：整体架构与防作弊原则

**1.1 架构图（逻辑关系）**

```
[玩家手机浏览器] ←→ [Railway 服务器]
  (前端 Canvas)        (后端 Node.js)
       |                    |
  只负责：             全权负责：
  - 画面渲染           - 所有数值计算
  - 音效播放           - 碰撞与合法性验证
  - 触屏操作采集       - 随机数生成
  - 向服务器发指令     - 状态广播
```

**1.2 防作弊铁则**

**前端永远是不可信任的。以下所有逻辑必须在后端实现并验证：**

· 伤害公式计算
· 装备合成验证（材料扣除与成品生成）
· 卡牌拾取、丢弃、组合交换
· 摇杆移动合法性（速度校验、穿墙检测）
· BOSS AI 决策
· 偷家磨损数值
· 所有随机数（暴击、掉落、刷新）
· 货币增减、信誉分变动

前端只负责发送“我往左走”、“我攻击了目标X”、“我拾取了物品Y”这样的意图指令，后端验证后广播结果给所有客户端。

**补充规则：**

· 移动合法性校验：后端每 tick 对玩家位置进行服务器端物理步进，检测是否穿墙或超速，若发现非法位置则强制回滚至上一合法坐标。
· 指令去重：所有客户端指令需附带序列号（seq）或时间戳，后端对 200ms 内同类型、同参数的重复指令进行去重，防止网络抖动导致攻击或拾取被多次执行。

**2.2 移动端适配要求（务必严格遵守）**

**① 视口设置**

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover">
```

禁止缩放，适配全面屏安全区。

**② 安全区适配**
所有交互按钮（摇杆、攻击键、卡牌栏）必须避开 safe-area-inset-bottom 和 safe-area-inset-left/right。使用CSS变量：

```css
:root {
  --safe-bottom: env(safe-area-inset-bottom, 0px);
  --safe-left: env(safe-area-inset-left, 0px);
  --safe-right: env(safe-area-inset-right, 0px);
}
```

摇杆的 left 值至少为 var(--safe-left) + 10px。
普攻键的 right 值至少为 var(--safe-right) + 10px。
卡牌栏的 bottom 值至少为 var(--safe-bottom) + 5px，且当屏幕宽度不足时，卡牌尺寸和间距会自动缩小，确保不遮挡安全区（见 2.4 卡牌栏弹性缩放规则）。

**③ 分辨率适配**

· 游戏设计基准分辨率：812 × 375（iPhone X 横屏逻辑像素）
· 所有元素尺寸使用 rem 或 vw/vh 单位，相对基准分辨率等比缩放。
· Canvas 画布实际分辨率取 window.devicePixelRatio * 逻辑尺寸，保证高清屏清晰。

**④ 热区设计**

· 所有可点击元素的最小热区为 44×44 CSS像素（符合iOS/Android人机交互规范）。
· 视觉图标可以更小，但透明热区必须覆盖满44×44。

**动画与丝滑要求**

· 60fps 硬指标：Canvas 主循环使用 requestAnimationFrame，任何一帧渲染不得超过16ms。
· 缓动曲线：所有UI动画使用 cubic-bezier(0.25, 0.1, 0.25, 1.0)（ease-out），拒绝生硬线性动画。
· 卡牌抽拉施法：卡牌跟随手指拖出时，使用 transform: translateY() 配合弹簧回弹动画，回弹曲线 cubic-bezier(0.34, 1.56, 0.64, 1)。
· 技能冷却光环：SVG 圆环从360度减至0度的动画，使用 stroke-dashoffset 动画。
· 击杀提示：从中央弹出，透明度从0到1再渐隐，带轻微上飘，总时长2秒。
· 界面切换：使用淡入淡出 opacity 过渡，时长0.25秒。

**渲染优化（补充）**

· Canvas 采用离屏分层：背景/网格层、实体层、粒子层、UI层分别绘制于独立离屏 Canvas，最后一次性合成到主屏，减少重绘开销。
· 使用脏矩形（Dirty Rectangle）技术，每帧仅更新视野内发生变化的部分。
· Service Worker 缓存策略：采用 Stale-While-Revalidate，并在 sw.js 中加入构建版本号校验，有新版时后台更新并提示用户刷新。

**3.2 核心模块职责**

**GameRoom（房间）**

· 管理一局游戏的完整生命周期
· 状态机：WAITING → EXPLORING → VOTING → BATTLE → FINISHED
· 持有10个Player实例、2个Boss实例、2个Base实例、1个Map实例
· 维护全局计时器（探索10分钟、选边各30秒）
· 每帧（每秒20次tick）广播状态给所有客户端

**Player（玩家）**

· 属性：血量、移速、攻击、防御、等级、Miss概率、剩余复活次数
· 装备：头盔、护甲、护腿、靴子、主动武器、被动武器
· 卡牌：最多5张卡牌数组
· 状态：存活/死亡/观战/离线
· 位置：x, y坐标（后端权威）

**Combat（战斗计算）**

· 伤害公式实现：
  ```
  最终伤害 = (攻击力² / (攻击力 + 目标防御力)) × 武器系数 × 随机波动(0.95~1.05)
  ```
· Miss判定：仅对普攻，实际闪避率 = Miss概率 × 50%
· BOSS对玩家伤害：200 + 递增（最多+100×5）  (已调整，见第六部分)
· 偷家磨损：按人数衰减表计算

**CardSystem（卡牌系统）**

· 拾取：验证距离，满槽时随机替换一张掉地
· 组合交换：验证两人距离，双方同意后执行交换流程
· 属性效果：水浸湿、火燃烧、土减伤、风冷却缩减
· 共鸣判定：武器主元素≥3且与卡牌一致
· 硬性冷却下限：无论元素点多少，任何卡牌最短冷却时间不得低于 2 秒（即原冷却的20%），防止无限连技。

**Crafting（制作系统）**

· 完整配方表（材料→半成品→成品）
· 验证玩家在工作台附近
· 扣除材料，生成装备，广播结果

**Economy（货币系统）**

· 结算公式：10金币 × 赔率倍率
· 天陨币合成：100金币可合成1天陨币，操作入口位于主界面背包中，要求信誉分≥60，且每局限合成1次。
· 7天/30天有效期定时检查：使用 UTC 时间戳对比，服务启动时及每小时执行一次过期扫描，不依赖本地零点。

**CreditSystem（信誉分）**

· 初始100分
· 3分钟无操作-20，主动退出-15
· 正常完成一局+5
· 10个玩家点赞 +1
· <30分 隔离匹配池
· 回归机制：信誉分低于100时，每正常完成3局比赛（无挂机/早退）恢复1分，直至100分为止。

**3.3 数据库设计**

使用 SQLite（轻量级，适合 Railway 部署）。所有写操作通过内存队列批量写入，每 500ms 提交一次事务，避免多玩家同时结束时发生锁竞争。

**players 表**

字段 类型 说明
id TEXT PRIMARY 账号ID（手机号hash）
nickname TEXT 昵称
gold INTEGER 金币数
god_coin INTEGER 天陨币数
gold_expire INTEGER 金币过期时间戳
god_coin_expire INTEGER 天陨币过期时间戳
credit_score INTEGER 信誉分，默认100
total_games INTEGER 总对局数

**3.5 服务器Tick速率**

· 游戏逻辑Tick：每秒20次（50ms间隔）
· 广播频率：
  · 移动：每秒10次（位置），客户端插值补帧
  · 战斗事件：即时发送
  · 经济/状态变化：即时发送
· 心跳检测：每5秒服务器发送 ping，客户端回复 pong，超时10秒判定离线

**3.6 断线/重连处理**

· 检测到断线：开始3分钟倒计时
· 3分钟内重连：恢复玩家状态，继续游戏
· 3分钟未重连：踢出对局，装备/卡牌掉落在地上，扣除20信誉分
· 对局中所有玩家立即收到 player_left 消息，该玩家位置生成掉落物
